# v1.1.0 — Telegram MTProto + Skill Passthrough + Model Control

## What was built

### Telegram user account login (MTProto)
Previously only Bot mode (requires @BotFather). Added user account mode via gramjs:
- `src/telegram-user-auth.ts`: QR login flow using `signInUserWithQrCode`
- `src/adapters/telegram-user.ts`: `TelegramUserAdapter` — receives private messages, sends replies
- `imcc telegram-login` CLI command
- 2FA (two-step verification) support
- Both modes can run simultaneously from the same config

QR code URL format: `tg://login?token=${token.toString('base64url')}`

gramjs typing issues encountered:
- `client.session.save()` is typed as `void` on the base class — kept a named `stringSession` variable typed as `StringSession & { save(): string }`
- `e.isPrivate` is `boolean | undefined` — filter uses `=== true`
- `password` callback signature uses `_hint?: string` (optional) not `_hint: string`

### New commands
- `/model [name]` — per-session model override, resets conversation. Model stored in `Session.modelOverride`, takes precedence over `config.claude.model`.
- `/doctor` — spawns `claude doctor` subprocess, returns stdout+stderr inline.
- `/skills` — lists SKILL.md files from `~/.claude/skills/` and `.claude/skills/`.
- `/<skill> [args]` — reads the skill's SKILL.md, passes it as the `-p` prompt. Args appended after skill content.

### Skill passthrough approach
Considered PTY (pseudo-terminal) to run skills interactively. Rejected because:
- PTY is OS-specific, fragile to parse
- `AskUserQuestion` tool calls inside skills would block waiting for terminal input
- No good way to forward the IM user's replies back to an interactive PTY over IM

Settled on prompt injection: `claude -p "${skill.content}\n\n${userArgs}"`. Trade-off: skills that use `AskUserQuestion` — Claude self-decides rather than prompting the IM user. Acceptable for now.

Skill lookup order (`src/skill-loader.ts`):
1. `<cwd>/.claude/skills/<name>/SKILL.md`
2. `<cwd>/.claude/skills/<name>.md`
3. `~/.claude/skills/<name>/SKILL.md`
4. `~/.claude/skills/<name>.md`

### Bug fixes
- **Feishu startup race**: moved `bridge.start()` before the adapter start loop in `cli.ts`. Feishu's WSClient can fire events during the connect handshake — handlers must be registered first.
- **`rl.close()` in claude.ts**: readline interface was not closed after process exit, causing a fd leak on long-running sessions.

## Tests added
`src/__tests__/session.test.ts`, `bridge.test.ts`, `claude.test.ts` — 24 tests total using vitest.

## What's NOT done yet (possible v1.2+)

- Streaming responses: currently waits for full Claude response before replying. Long tasks (e.g. multi-file edits) give no feedback until done. Could send "thinking..." or stream partial results.
- Image/file support: IM platforms can send images; Claude Code can handle them. Not wired up.
- Session persistence across `imcc` restarts: `claudeSessionId` is in-memory only. Restart loses conversation context.
- Webhook mode for Feishu/Telegram: current polling/WebSocket requires outbound connection from the machine. Could add webhook mode for server deployments.
- Push notifications: currently only responds to messages, no way for imcc to proactively send a message (e.g. "build finished").
