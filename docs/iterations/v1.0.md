# v1.0.0 — Initial Release

## What was built

Three IM platform adapters + a core bridge connecting them to Claude Code CLI.

**Platforms:**
- Telegram (Bot mode via Telegraf)
- Feishu / Lark (WebSocket long-connection via `@larksuiteoapi/node-sdk`, QR login via `feishu-auth.ts`)
- WeChat (iLink HTTP long-polling, QR login built in)

**Commands:** `/new`, `/cwd [path]`, `/info`, `/help`

**Core features:**
- Multi-turn conversation via `--resume <session_id>`
- Per-user isolated sessions (cwd, claudeSessionId)
- Per-user message queue (no concurrent Claude processes per user)
- WeChat message dedup (in-memory Set, 200 message limit)
- Long message splitting (`splitMessage` in utils.ts, shared across adapters)
- `allowedUsers` whitelist in config

## Key decisions made

**Why `claude -p` and not Claude API:** We want full tool access (file editing, terminal, MCP), not just text generation. The CLI is the only way to get that.

**Why WebSocket for Feishu and not webhooks:** Webhooks require a public HTTPS endpoint. WebSocket long-connection works from any machine with outbound internet. Same philosophy as WeChat's polling approach.

**Why split `splitMessage` into `utils.ts`:** All three adapters had identical chunking logic. Extracted to eliminate duplication.

## What was borrowed

- WeChat adapter: protocol and architecture ported from [weclaw](https://github.com/fastclaw-ai/weclaw) (MIT). Translated Go → TypeScript. The iLink API itself (`ilinkai.weixin.qq.com`) is WeChat's, not weclaw's.
- Feishu auth flow: reverse-engineered from `@larksuite/openclaw-lark-tools` (ISC). The OAuth device registration endpoint (`/oauth/v1/app/registration`) is not documented publicly.
- imclaw / Claude-to-IM: architecture inspiration only, no code.

## Known issues / punted to later

- Feishu startup race: if a message arrives during the WebSocket handshake before `bridge.start()` registers handlers, it's silently dropped. Fixed in v1.1.
- No `/model` command — model is config-only.
- No skill support.
